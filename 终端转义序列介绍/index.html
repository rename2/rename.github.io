概述#

当我们日常登录机器，无论是SSH、RDP还是VNC连接，我们都在使用终端。终端(Terminal)是一种输入输出设备，用于人类用户和计算机的交互。其中最常见的终端类型是面向字符的终端，也就是每次与主机通信一个字符，而不是一个数据块。字符可以是控制字符，也可以是a-zA-Z0-9这些可打印字符。一串字符组成了字符序列，可以用于控制终端和字体颜色、光标位置、切换输入模式（如替换模式）等，这样的字符序列称为转义序列。

根据对控制字符和转义序列的支持程度不同，面向字符的终端被分为“哑终端”和“智能终端”。哑终端(dumb terminal)只能解释有限的控制字符（如CR、LF等），但无法处理清除行、移动光标等转义序列。智能终端(smart terminal)是有转义序列处理能力的终端，支持VT100、ANSI等转义序列。

转义序列标准，如VT52，来源于物理硬件终端。这些终端使用供应商特定的转义序列处理比如光标放在屏幕上指定位置的操作，虽然功能相同，但序列不同。程序也不得不根据终端的型号发送对应的转义序列，以保证终端的正常显示。于是ANSI标准出现了，它制定了所有终端都采用的命令集，并且要求所有数字信息作为 ASCII 数字传输。VT100终端最先支持了ANSI序列，并由于在市场上的成功成为了后来终端仿真器事实上的标准。

之后个人计算机开始普及，软件可以在PC上模拟、而不用增设昂贵的硬件终端。这种模拟终端的计算机程序被称为终端模拟器（终端仿真器）。xterm是X Window 系统的标准终端仿真器。X Window 系统是用于位图显示的窗口系统，常见于类Unix操作系统上（如Darwin，macOS的核心）。因为X协议一直处于版本11的状态，也称为X11。SSH中的X11 forwarding也是对于X Window的转发支持。xterm支持ANSI、VT52、VT100等控制协议，还添加了一些其他终端仿真器已采用的协议，如16色终端协议、88色终端协议和256色终端协议。打开终端、查询环境变量TERM，可能看到值为“xterm-256color”，表示采用xterm的256色终端协议。

ANSI标准#

单字节字符也可以控制终端，这些字符称为C0控制代码(C0 control codes)，由ASCII 0到31共32个控制字符组成，例如

  • BEL（0x7）：发出蜂鸣
  • BS （0x8, \b）：退格，向左移动光标
  • LF（0xA, \n）：换行
  • CR（0xD, \r）：将光标移到第0列
  • ESC（0x1B）：启动转义序列

所有转义序列都以ASCII字符ESC开始，第二个字符指定对应的子序列或命令。其中第二个字符范围为@A-Z[\]^_的序列称为C1控制代码(C1 control codes)，例如

  • ESC [：CSI(Control Sequence Introducer)，大部分常用序列都是CSI
  • ESC ]：OSC(Operating System Command)，操作系统使用的控制序列，比如窗口标题、设置调色板颜色
  • ESC c：重置为初始状态，包括图形格式、默认字体等

Control Sequence Introducer(CSI)#

以ESC[（也可以写成\e[ 或者 033[）为前缀，CSI序列的基本格式为parameter intermediate final。

  • parameter：0–9:;<=>?中的字符，表示控制操作的参数，比如删除的字符数量
    • 可以为空，序列会设置自定义的默认参数，比如默认为1、删除1个字符
    • 也可以是分号分隔的数字，如1;2;3，中间缺失的数字视为0，比如1;;3等同于1;0;3，设置显示属性序列中常用（设置字体为粗体、前景色、背景色等）
  • intermediate：!"#$%&'()*+,-./中的任意数量字符，大部分序列不用
  • final：@A-Z[\]^_`a–z{|}~中的单个字符，表示操作，比如K表示清除本行字符(Erase in Line)

CSI示例

  • CSI n C：光标向前移动n个单元格
  • CSI n G：移动光标到第n列（n默认为1）
  • CSI n K：清除本行字符（但不移动光标），如果n为0，清除从光标到行尾的内容；如果n为1，清除从行首到光标的内容；如果n为2，清除整行

Select Graphic Rendition(SGR)#

CSI中还包括一个可以设置显示属性的序列——CSI n m，称为Select Graphic Rendition (SGR)。 多个属性可以在同一个序列中设置，用分号分隔。每个属性设置后持续有效，直到再次用SGR序列重置。 SGR的n表示不同含义，例如

  • 0：重置或正常
  • 1：粗体或颜色增强
  • 2：细体或颜色减弱，比如用于打印debug信息
  • 3：斜体
  • 4：下划线
  • 7：反转前景色和背景色
  • 30-37：设置前景色，只有八种标准色可选
  • 38：设置前景色，支持更多颜色、支持RGB格式
  • 40-47：设置背景色，支持和前景色一样的八种标准色，只是代码不同
  • 48：设置背景色，支持更多颜色、支持RGB格式

最初只定义了八种标准色：

颜色 前景色 后景色
黑色 Black 30 40
红色 Red 31 41
绿色 Green 32 42
… … …
白色 White 37 47

这些颜色现在也经常直接使用，比如ESC[37;40m设置为白色字母、黑色背景，ESC[31m设置为红色，ESC[1;31m设置为亮红色，以及用ESC[0m重置所有属性。

当需要更多颜色，可以使用预定义的256色卡中的颜色，或者用RGB格式设置

  • ESC[38;5;n m：38表示设置前景色，5是固定参数表示使用色卡，n为颜色值，比如105表示蓝紫色
  • ESC[48;2;r;g;b m：48表示设置背景色，2是固定参数表示使用RGB格式，r、g、b分别设置为对应的数值

Operating System Command(OSC)#

另外一种序列是OSC，前缀是以ESC]，通常以数字开头，后面跟着分号和剩余部分，最终以终止符（\a和\）结束。 最常见的用法是设置窗口标题

  • OSC 0;{title}\a：设置窗口和标签页标题
  • OSC 1;{title}\a：设置标签页标题
  • OSC 2;{title}\a：设置窗口标题

转义序列的捕获#

既然很多操作实际由转义序列承载和完成，那怎么获取到对应的序列呢？以下代码将终端设置为raw模式，直接读取原始的序列。

package main

import (
    "bytes"
    "fmt"
    "os"
    "os/exec"
    "sync/atomic"

    "github.com/creack/pty"
    "golang.org/x/term"
)

func main() {
    err := run()
    if err != nil {
        fmt.Println("err:", err)
        os.Exit(1)
    }
}

func run() error {
    os.Setenv("TERM", "xterm-256color")

    c := exec.Command("/bin/sh")
    pty, err := pty.Start(c)
    if err != nil {
        return err
    }
    defer func() { _ = pty.Close() }()

    fds := []int{int(os.Stdin.Fd()), int(os.Stdout.Fd())}
    for _, fd := range fds {
        fd := fd
        oldState, err := term.MakeRaw(fd)
        if err != nil {
            return err
        }
        defer func() { _ = term.Restore(fd, oldState) }()
    }

    typing := atomic.Bool{}
    typing.Store(false)

    go func() {
        buf := make([]byte, 512)
        sequence := make([]byte, 0, 1024)
        for {
            n, err := os.Stdin.Read(buf)
            if err != nil {
                return
            }

            sequence = append(sequence, buf[:n]...)
            if bytes.ContainsAny(sequence, "\r\n") {
                typing.Store(false)
                fmt.Printf(debugFormat("\r\nIn : %q\r\n"), sequence)
                sequence = sequence[:0]
            } else {
                typing.Store(true)
            }

            _, err = pty.Write(buf[:n])
            if err != nil {
                return
            }
        }
    }()

    buf := make([]byte, 512)
    sequence := make([]byte, 0, 1024)
    for {
        n, err := pty.Read(buf)
        if err != nil {
            return nil
        }

        sequence = append(sequence, buf[:n]...)
        if !typing.Load() {
            fmt.Printf(debugFormat("Out: %q\r\n"), sequence)
            sequence = sequence[:0]
        }

        _, err = os.Stdout.Write(buf[:n])
        if err != nil {
            return nil
        }
    }
}

func debugFormat(str string) string {
    return fmt.Sprintf("\x1b[0;2m%s\x1b[0m", str)
}

尝试不同的输入和快捷键，可以看到对应的输入和输出序列。再结合前面提到的转义序列格式，可以推断出最终效果。

转义序列的应用#

最常见的使用方式是显示属性的设置，比如在ls --color中，就分别为目录、可执行文件、软链和普通文件等设置了不同的颜色。 使用dircolors可以查看当前的颜色设置，也可以通过环境变量自定义颜色设置，比如将目录颜色设置为黄色（原来的目录设置为01;34，即颜色增强的蓝色）。

当脚本打印日志时，也可以使用序列突出显示错误日志。前面的Go代码中打印debug信息就用了转义序列。

print_error(){
    printf "\e[1;31m$1\e[0m\n"
}

如果想实现一个自己的终端模拟器，或者截取了服务器端流量、想要分析对应的执行命令，则需要尽可能多地识别和解析常见序列，才能得到正确的命令内容、实现交互效果。

  • golang.org/x/term库支持启动VT100终端、读取相关输入。
  • darktile实现了一款自定义的终端模拟器，对转义序列支持良好，很有参考价值。
